Range Slider & Drop Zeroes — Component Design Review
UX improvements to the Data Card visualization panel — covering Range Slider behavior, Drop Zero toggle, filter panel layout, and engineering handoff readiness. Key decision: Drop Zero and Range Slider are fully independent filters.
Recording
Discussion Notes
Keep Visualization Panel Open After Apply
Current behavior: clicking Apply closes the visualization panel automatically.
Problem: user must reopen, adjust, and re-apply — unnecessary extra clicks and poor usability for iterative adjustments.
Decision: keep panel open after Apply; add a manual Close (X) button so users control when to dismiss.
Updated Filter Panel Layout
New layout: labels on the left side, actions/controls on the right side.
Drop Zero → Toggle on the right
Range → Slider + Input fields on the right
Opacity → existing behavior retained
Benefits: cleaner scanning, better alignment, more organized UI that scales with more filter types.
Drop Zero Toggle Behavior
Default state: Drop Zero toggle is ON by default — users normally don't want zero-value cards visible on the map.
When Drop Zero is ON: zero-value cards are hidden from the map — slider range is NOT redefined.
Drop Zero only filters zero values; it does not change or move the slider endpoints.
Drop Zero and Range Are Independent (Key Decision)
Earlier assumption: turning Drop Zero ON would move slider minimum to zero.
Final decision: do NOT connect them.
Range filter controls which value range is displayed; Drop Zero only hides zero values within that range.
Example: Range set to -30 to +30 with 50 cards; if 20 cards are zero-value and Drop Zero is ON, 30 cards shown. Slider stays at -30 to +30.
Negative numbers are supported — Drop Zero must work correctly when values span negative to positive ranges.
Range Input Text Fields
Phase 2 addition: add numeric text fields for minimum and maximum alongside the slider.
Default values should come from actual API data range (e.g., if Gas Storage API returns range 20–50, UI shows 20 to 50, not 0 to 500).
Input validation: block alphabetic input via HTML5 validation — remove custom error messages (HTML5 handles this natively).
Filter Panel Height & Reset
Remove internal scrollbar — make panel height dynamic (more filters = panel grows; fewer filters = panel shrinks).
Reset button added and appreciated — with multiple filters, reset is important for usability. Position: top right area alongside Close button.
Scope Decision
In scope: Drop Zero toggle, Range Slider improvements, text input for range, dynamic panel height, new toggle styling, minor layout updates.
Out of scope: fill opacity changes, larger visualization redesign — planned separately based on priority.

UX Insights
Iterative Filter UX
Closing a panel on Apply forces unnecessary reopens during fine-tuning. Panels used for iterative work should stay open until the user is done.
Separate close and apply is a standard pattern for power-user filter panels.
Filter Independence Principle
Linking Drop Zero to the Range Slider would create invisible side effects — users would move one control and see another change unexpectedly.
Independent filters are predictable and easier to reason about, especially when debugging why data appears/disappears.
Data-Driven Defaults
Slider range defaulting to actual API data range (not 0–500) gives users an immediately meaningful starting point.
Generic 0–500 defaults force users to hunt for their real range, adding friction before any work starts.
Lean on Platform Validation
HTML5 input validation natively prevents non-numeric input. Custom error messages for the same scenario are redundant and add maintenance overhead.
Save custom validation effort for rules the platform cannot handle (e.g., min > max).
