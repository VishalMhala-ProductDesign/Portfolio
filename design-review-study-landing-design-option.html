Design Review — Recording
Study Landing Page & TAG — Design Option
Comparison of two landing page design options for PLEXOS Cloud — left-navigation-based vs. compact dashboard. Final conclusion: Option 2 has stronger UX potential, but the information architecture should follow the simplified study categorization established from Option 1.
Recording
Design Options Compared
Option 1
Left Navigation Based
Persistent left nav with study categories: My Studies, All Studies, Favorites, Archive, Tags. Provides clear IA structure but can feel heavy for a focused study management task. Main contribution: strong, simplified information architecture adopted for final direction.
Option 2 — Preferred
Compact Dashboard
More compact, dashboard-style layout. Key actions (Create Tag, Archive, Favorites, Activity) accessible from top right. Received more positive feedback — better balance of density and usability. Final direction: adopt Option 2's visual approach with Option 1's IA structure.
Discussion Notes
Information Architecture — Final Study Sections
My Studies: all studies the user has access to; includes recent indicator inline (not as a separate section); shows favorite and status indicators.
All Studies: every study in the tenant including accessible, non-accessible, public, private; user filters within rather than separate tabs.
Favorites: user-marked studies with star icon behavior.
Archive: shows archived studies — kept as a separate section.
Studies
├── My Studies
│    └── Recent indicators inline
├── All Studies
│    ├── Filter: Has Access / No Access
│    └── Filter: Public / Private
├── Favorites
└── Archive

Tags
├── User-created tags
└── Default tags (Desktop, CLI, Marketplace)
Items Removed from Navigation
No Access Studies removed as a top-level nav category — it is a filter inside All Studies.
Delete section removed — deleted studies are permanently removed; no recycle bin. Recommended flow: Archive first, then delete permanently.
Messages removed from nav — comments/activity are actions on studies, not categories. Should appear inside study context instead.
Recent Studies removed as a separate section — merged into My Studies with inline recency indicator.
Tenant Switching
User belongs to one primary tenant/company; tenant switching possible without logout — switching updates all data instantly.
Backend supports primary + secondary tenant. Tenant selector on top needs improvement to make the tenant card more visible.
Study Card Information
Required fields: Study title, Description, Tags, Source software (Desktop/CLI/Marketplace), Access type, Favorite indicator, Comments/activity indicator.
Source software shown as a chip/tag — helps users understand where the study originated.
Public Study: show public icon; do not show full user list. Private Study: show assigned users and access information.
Tagging — Simplified Flow
When user creates a tag from a study context, automatically assign it to that study — user can remove if needed.
Flow: Type tag name → Create → Automatically assigned. Tag management: Create, Assign, Remove. No delete from this flow.
Possible tag maximum: 20–50 tags. Design must handle overflow with "+3 tags" collapsed view.
Open Study Behavior
Concern: if study opens in a new tab, user may accidentally close the landing page and lose their place.
Reference pattern: Figma/cloud drive — user can return to the file list from inside the open study.
Possible solution: keep landing page accessible; add Home/Landing navigation inside the open study environment.
Open Study button feedback: current button too small and has insufficient visual hierarchy — needs size, placement, and primary action styling review.
UX Insights
IA First Principle
Defining the correct information architecture before visual design prevents edge cases, state issues, and messaging problems from surfacing late in the process.
Navigation structure shapes every interaction that follows — locking this early reduces downstream redesign risk.
Filters Over Nav Categories
Replacing "No Access Studies" nav item with a filter inside "All Studies" simplifies the nav without hiding the functionality.
Adding more top-level nav items for every edge case creates navigation bloat. Filters scale better than nav expansion.
Auto-Assign Tags on Creation
Creating a tag from a study context and automatically assigning it removes a redundant step — the user's intent is to tag this study, so why make them do it manually?
Default-assign with easy removal is a lower-friction pattern than requiring an explicit assign step.
Navigation Continuity After Open
If users lose the landing page by opening a study, they lose their browsing context — especially costly after applying filters or sorting.
A persistent "back to studies" path inside the open study (like Figma's header link) solves this without forcing multi-tab management on the user.
