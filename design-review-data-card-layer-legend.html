Data Card, Layer & Legend
End-to-end workflow validation for Map Layering, Data Cards, Properties Panel interactions, and Show/Hide behavior across layers. The primary unresolved issue is achieving a single source of truth for layer visibility when users can toggle layers from two different panels.
Recording
Discussion Notes
Layer Positioning & Floating Data Card
Data Card always stays on the top/floating layer regardless of other visualization switches.
Changing from Data Card to Bubble keeps Data Card fixed at the top position; switching visualization types updates layer order automatically.
Avoids users needing to manually re-position the Data Card repeatedly after every visualization change.
Pinning & Editing Capabilities
Existing pinning functionality unchanged — users can hover, open the Edit panel, pin items, and clear all pins.
Editable properties: Value, Unit, Property object name.
Layer Ordering Behavior
Selecting a property from the Properties Panel moves that item to the top of the Map Layering panel.
Moving a property up/down in the layer order updates the Data Card displayed value accordingly.
Example: moving Max Volume below Injection changes the card value to show Injection.
Layer Visibility (Show/Hide) — Two Different Behaviors
Hiding from Properties Panel: layer disappears completely from the Map Layering panel.
Hiding from Map Layering Panel: layer hides visually but remains available in the layer list — user can re-enable without returning to Solution Panel.
Inconsistency between the two creates confusion and makes behavior hard to predict.
Single Source of Truth Concern
Two different states tracking the same toggle creates UX inconsistency and potential engineering bugs.
Suggested direction: maintain a single state/source of truth for visibility, regardless of where the user clicked.
Concern flagged to discuss with Chetan (developer) and Tim for technical feasibility.
If Map Layering hide removes the layer entirely, user must return to Solution Panel to re-enable — poor UX. Preferred: keep layer accessible in Map Layering as a persistent chip, only visually hidden.
Visualization Panel Placement — Open Decision
Discussion: Should visualization type controls exist only in Properties Panel, only in Map Layering, or both?
Emerging direction (Chetan's suggestion): remove visualization selection from Properties Panel and keep it only in Map Layering — needs confirmation.

UX Insights
State Consistency Principle
When the same action (hide a layer) can be triggered from two different places, both must produce the same result and reflect the same state.
Two-source state tracking is a UX and engineering anti-pattern — always resolve to a single source of truth.
Recovery Path Design
If hiding a layer requires the user to navigate to a completely different panel to undo, the hide action is too destructive for its intent.
Prefer soft-hide (persistent chip, visually toggled off) over hard-remove from the layer list.
Floating Priority Layer
Data Card's auto-float to the top layer removes a class of user errors — users don't have to manually manage priority after every switch.
Auto-ordering based on layer type is a pattern worth applying to other visualization hierarchies.
