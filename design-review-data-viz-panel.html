Design Review — Recording
Data Visualization Panel on PRO Panel & Layering Panel
Three design options compared for data visualization controls across the Property Panel and Map Layering Panel. Final decision: Option 2 confirmed — visualization controls available in both panels. Data Card always floats above all other visualizations.
Recording
Design Options Compared
Option 1
Properties Panel Only
User selects visualization from Property Panel; editing opens in Layering Panel. Interaction feels disconnected across two separate panels.
Option 2 — Selected
Both Panels
Existing visualization panel remains in Property Panel; clicking visualization opens the panel normally. Already implemented, familiar, faster development, less user education needed.
Option 3
Hover-Based
Hover over property → quick visualization controls appear. Concern: constant hover menus feel intrusive when user is just selecting/unselecting properties. Possible mitigation: hover delay.
Discussion Notes
Map Layering Workflow
User hovers over a layer/property in Map Layering → quick action menu appears.
User can: open visualization controls, switch visualization type, reorder layers.
Switching visualization updates the map immediately; change reflects simultaneously in Layering Panel, Property Panel, and Data Visualization Panel.
Data Card Layering Behavior
Key rule: Data Card always floats on top of all visualizations.
If a property changes to Data Card → it automatically moves above other visualizations.
If changed back to another visualization type → returns to normal ordering.
A separator line in the Map Layering panel defines the boundary between the Data Card section and the Other visualization section.
Data cards cannot be placed inside the normal visualization section and vice versa — system auto-corrects forced repositioning.
Layer Shuffling & Multiple Properties
Users reorder layers using drag handles — map updates immediately on reorder. Applies to point layers, line layers, and data cards.
Point objects and line objects (e.g., pipelines) follow the same layering behavior and separator line rule.
In unpinned state: only the topmost property value is visible on the Data Card.
When user pins the Data Card: all stacked property values become visible (follows existing data card behavior).
Opacity & Visual Settings
Opacity control exists in visualization settings; Apply button appears only after a modification is made.
Users are not expected to adjust opacity/size frequently — possible future move to user preferences panel.
Bubble size is calculated from the property value, not a fixed setting — Small/Medium/Large controls likely unnecessary and may be removed.
Hide/Show Behavior — Still Open
Current proposal: hiding a property removes it from Map, Property Panel, and Layering Panel everywhere.
Technical feasibility must be confirmed with Shivam/Chetan before finalizing this behavior.
UX Insights
Why Option 2 Won
Already implemented — reduces engineering risk and timeline.
Familiar pattern — users already know how to use it, so onboarding cost is near zero.
Hover-based patterns (Option 3) introduce discoverability risk — users may never find controls they can't see.
Auto-Layer Priority Rule
Automatically placing Data Card above all other visualizations is a smart default that removes a whole class of "why is my card hidden?" support issues.
Explicit separator line in the UI makes the rule visible and learnable, rather than hidden magic.
Real-Time Sync Across Panels
When switching visualization types, all three panels (Map, Layering, Property) must update simultaneously.
Out-of-sync panel state is a major trust issue — users lose confidence in what is "actually" happening on the map.
